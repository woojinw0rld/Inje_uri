## §2. 3단 분리 검증

### 2-1. Route (thin shell)

| # | 체크 항목 | 통과 |
| --- | --- | --- |
| 1 | Route에 비즈니스 로직이 **없는가** (if 분기, DB 판단 등 금지) | [ ] |
| 2 | 요청 파싱(body, params, query)만 하고 Service 함수 호출 | [ ] |
| 3 | 성공 시 `ok(result)` 반환 | [ ] |
| 4 | 실패 시 `catch(e)` → `e instanceof AppError` → `fail(e.code, e.message)` | [ ] |
| 5 | Service/Repository를 직접 import하지 않고 **Service만** import | [ ] |

### 2-2. Service (비즈니스 로직)

| # | 체크 항목 | 통과 |
| --- | --- | --- |
| 1 | 비즈니스 판단(권한, 유효성, 상태 체크)이 **Service에만** 있는가 | [ ] |
| 2 | 에러 시 `throw new AppError(ERROR.XXX, "메시지")` 사용 | [ ] |
| 3 | DB 결과(snake_case) → 응답(camelCase) 변환이 Service에서 수행되는가 | [ ] |
| 4 | 트랜잭션이 필요한 경우 Service에서 `prisma.$transaction()` 시작하는가 | [ ] |
| 5 | Repository만 호출하는가 (다른 Service 직접 호출 X, Prisma 직접 호출 X) | [ ] |

### 2-3. Repository (DB 쿼리)

| # | 체크 항목 | 통과 |
| --- | --- | --- |
| 1 | Prisma 쿼리**만** 있는가 (비즈니스 판단 X, 데이터 변환 X) | [ ] |
| 2 | 반환값이 Prisma 네이티브 타입인가 (camelCase 변환 안 함) | [ ] |
| 3 | 트랜잭션 필요 시 `tx` 파라미터를 받아서 사용하는가 | [ ] |

| --- | --- | --- | --- |
| Route | ① 요청 파싱 | body/params/query 추출 | — |
| Service | ② | (비즈니스 검증 1) | `에러코드` — 메시지 |
| Service | ③ | (비즈니스 검증 2) | `에러코드` — 메시지 |
| Repository | ④ | (DB 쿼리) | — |
| Service | ⑤ | (DTO 변환) | — |
| Route | ⑥ 성공 응답 | `ok(result)` → `{ success: true, data: { ... } }` | — |
