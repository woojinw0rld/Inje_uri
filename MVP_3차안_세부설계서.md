# 인제우리 MVP 3차안 — 세부설계서 기준 재정의본

> **기준선:** 최신 UI/UX 요약 + 실제 화면 프로토타입 + schema.prisma + migration.sql + A/B/C/D 확정 정책
>
> **해석 원칙:** 프론트는 mock data 기반, Prisma/migration은 구현보다 앞선 구조
>
> **중요:** MVP 범위와 장기 설계를 절대 혼합하지 않음

---

## 1. 문서 목적과 적용 범위

이 문서는 현재까지 확정된 세부 정책과 최신 스키마를 기준으로, **실제 출시 가능한 범위만 추려낸 MVP 3차안**을 정의한다.

목적은 기능을 더 추가하는 것이 아니라, **현재 구현해야 할 범위를 명확히 잠그고** 프론트 / 백엔드 / API 계약의 기준선을 제공하는 데 있다.

> MVP 기준 문서다. 장소 추천, 전역 알림 스트림, 전화번호 기반 차단, 운영 자동화처럼 스키마에는 있거나 정책 논의가 일부 존재하지만 아직 비용 대비 우선순위가 낮은 항목은 **후속 범위로 분리**한다.

---

### 1-1. Source of Truth

| 우선순위 | 기준 | 해석 |
|----------|------|------|
| 1 | 현재 확정된 A/B/C/D 정책 | 행동 규칙과 예외 처리의 최상위 기준 |
| 2 | schema.prisma / migration.sql | 실제 구현 가능한 데이터 구조 기준 |
| 3 | UI/UX 요약 및 실제 화면 파일 | 화면 의도와 사용자 흐름 기준 |
| 4 | 이전 초안 / 폐기 논의 | 충돌 시 무시 |

---

### 1-2. 현재 상태 진단

- 프론트는 mock data 기반 UI 프로토타입이며, 화면 intent는 구체적이지만 실제 데이터 연동은 아직 없다.
- DB / Prisma / migration은 이미 상당히 구체적이며, 일부 정책은 오히려 구현보다 앞서 있다.
- 따라서 MVP 3차안의 목적은 새 기능 설계가 아니라 **이미 합의된 기능 중 지금 출시 가능한 범위를 확정**하는 것이다.

---

## 2. MVP 3차안 한 줄 정의

> 대학생 전용 모바일 매칭 서비스에서 **하루 추천 3명**, **호감 수락 후 시간제한 채팅**, **짧게 노출되는 피드와 댓글 기반 상호작용**만 우선 구현하는 1차 출시안.

---

## 3. MVP 3차안 핵심 목표

- 추천 → 호감 → 채팅으로 이어지는 **주력 플로우**를 실제 API로 완결한다.
- 피드 → 댓글 → 선택 시 채팅으로 이어지는 **보조 플로우**를 최소 범위로 구현한다.
- 차단 / 신고 / 벤의 **공통 정책**을 모든 도메인에 동일하게 적용한다.
- 프론트 mock 상태를 실제 API 타입과 상태값으로 교체할 수 있는 수준까지 **계약을 고정**한다.

## 4. MVP 3차 포함 범위

### 4-1. 필수 포함 기능

| 영역 | MVP 3차 포함 기능 | 비고 |
|------|------------------|------|
| 인증/회원 | 로그인, 세션 확인, 학교 이메일 인증 최소 버전, 내 정보 조회 | 회원가입은 기존 인제 정보 시스템 연동 전제 |
| 프로필 | 프로필 수정, 프로필 이미지, 한 줄 소개, 프로필 키워드 / 이상형 키워드 수정 | 닉네임 중복 불가 |
| 추천 | 매일 09:00 추천 3명 생성, 하루 1명 선택, 패스 | `onboarding_completed`는 추천 필터에서 사용 안 함 |
| 호감 | 받은 호감 / 보낸 호감 조회, 수락, 거절 | 메시지 기능 제외 |
| 채팅 | 채팅방 생성, 목록, 상세, 메시지 전송/조회, 읽음 처리, 나가기, 만료 후 읽기만 | 실시간은 채팅방 내부 SSE만 |
| 피드 | 피드 목록, 상세, 작성, 수정, 키워드 필터, 최신순 정렬 | 1인 1 활성 피드 규칙 적용 |
| 댓글 | 댓글 작성, 내가 남긴 댓글 확인, 피드 작성자 선택 시 채팅 연결 | 1인 1댓글 규칙 적용 |
| 안전/운영 | 차단, 신고, 벤 사용자 비노출 / 로그인 차단 | 신고 시 차단 동시 선택 가능 |

### 4-2. MVP 3차 제외 범위 (Phase 2 / 후속)

| 영역 | 제외 기능 | 이유 |
|------|-----------|------|
| 채팅 | 1회 연장 API | DB 필드는 유지하되 MVP 구현 비용 대비 효과 낮음 |
| 채팅 | 장소 추천 API | schema 존재하지만 C 정책상 MVP 제외 |
| 알림 | 전역 알림 SSE | 채팅방 내부 SSE만으로 1차 출시 가능 |
| 차단 | 전화번호 기반 차단 | 정책은 확정됐으나 스키마 delta와 운영 검증 필요 |
| 운영 | 고객센터 재심사 프로세스 | 정책 존재하나 도메인 모델 미정 |
| 랭킹 | 신고 / 댓글 기반 고급 피드 랭킹 | 기본 최신순 + boost_score로 우선 출시 |
| 연락처 | 연락처 업로드 / 매칭 | DB만 존재, 제품 가치 검증 전 |
| 메시지 | 메시지 soft delete API | 필드는 있으나 MVP 미구현 |

## 6. 도메인별 세부 범위

### 6-1. A: 인증 / 프로필 / 공통 정책

- 닉네임 중복 불가, 변경 주기 제한 없음
- 탈퇴는 hard delete가 아니라 `withdrawn` / 프로필 비노출 방식으로 처리
- 차단 기록은 탈퇴 이후에도 유지
- 벤된 사용자는 로그인 불가 및 모든 노출 제외
- 차단 / 신고는 모든 관계(프로필 / 채팅 / 피드 / 댓글)에 동일 적용





## 7. 현재 정책과 스키마의 핵심 갭

| 항목 | 정책 상태 | 현재 구현 기준 | 처리 방안 |
|------|-----------|----------------|-----------|
| `onboarding_completed` | 추천 필터에서 삭제 확정 | `users` 테이블에는 존재 | 필드는 유지하되 추천 로직에서 무시 |
| 호감 메시지 | 삭제 확정 | 기존 UI mock에 흔적 존재 | 프론트 최신화에서 제거 |
| `blocked_by_user_id` | C 정책상 필요 | migration 미반영 | 스키마 delta 또는 서비스 레벨 보완 |
| 댓글 기반 채팅 출처 | 정책상 확정 | `source_comment_id` 미반영 | MVP는 서비스 레벨 추적, 추후 delta 권장 |
| 전화번호 기반 차단 | 정책 확정 | migration 미반영 | Phase 2 또는 schema delta 후 반영 |
| 피드 1인 1활성 | 정책 확정 | DB unique 없음 | 서비스 레이어 check |
| 피드 1인 1댓글 | 정책 확정 | unique 미반영 | 서비스 레이어 check 또는 partial unique 후속 적용 |
| 전역 알림 SSE | 미확정 | 구조 없음 | MVP 제외, 폴링으로 대체 |

## 8. MVP 3차 구현 순서 (Step-by-Step)

### Step 0. 기준선 잠금

- 현재 정책 문서, schema.prisma, migration.sql, UI/UX summary를 기준선으로 잠근다.
- 폐기된 초안은 모두 제외하고, 정책-스키마 갭 표를 공식 문서로 고정한다.

### Step 1. API 계약서 초안 작성

- A/B/C/D 별로 endpoint, request/response, auth, 에러 코드를 고정한다.
- 프론트는 이 문서를 기준으로 mock type을 실제 API 타입으로 교체한다.

### Step 2. 실행 구조 잠금

- 폴더 구조, env 변수명, Docker 기준, 포트 정책을 확정한다.
- B가 실행 구조 오너 역할을 맡는다.

### Step 3. 레포 통합

- schema / migration / API 계약서 / `.env.example` / Dockerfile / README를 기준 브랜치로 통합한다.

### Step 4. 프론트 최신화

- `/interest`에서 message 제거
- `/self-date`에서 좋아요 / 호감 CTA를 댓글 중심 CTA로 변경
- `/match` 상태값을 `selected_at` / `passed_at` 기준으로 변경

### Step 5. 백엔드 구현

1. A: auth / profile / report / block
### Step 6. 통합 QA

- 추천 선택 → 호감 목록 반영
- 호감 수락 → 채팅방 생성
- 피드 댓글 → 작성자 선택 → 채팅
- 차단 시 추천 / 피드 / 채팅 전부 제외

## 9. MVP 3차 승인 조건 (Acceptance Criteria)

| 영역 | 승인 기준 |
|------|-----------|
| 추천 | 매일 09:00 기준 오늘 추천 3명 생성, 1명 선택 잠금, 패스 1주 제외 적용 |
| 호감 | 받은 호감 수락 / 거절 가능, 무반응 호감은 유지, 거절 후 1주 쿨다운 적용 |
| 채팅 | 호감 수락 시 채팅방 생성, 24시간 만료, 만료 후 읽기 가능 / 쓰기 불가, unread 계산 가능 |
| 피드 | 1인 1활성 피드, 최신순 목록, 키워드 필터, 활성 피드에만 댓글 가능 |
| 댓글 | 1인 1댓글, 작성자 선택 시 채팅 연결 가능 |

| 프로필 | 프로필 / 이미지 / 키워드 / 추천 설정 수정 가능 |

## 10. MVP 3차 출시 리스크

1. **D 정책과 현재 self-date UI 불일치** — 피드 좋아요 / 호감 흐름을 댓글 중심으로 재정리하지 않으면 구현 중 혼선이 발생한다.

2. **C 정책의 `blocked_by_user_id`, `source_comment_id` 미반영** — API 설계 전 반영 여부를 결정해야 한다.

3. **무반응 호감 정책과 `interests.expires_at` 충돌** — B 정책상 무반응 호감은 무한 유지인데, 스키마에는 `interests.expires_at`가 있다. 자동 만료 배치를 잘못 넣으면 정책과 충돌한다.

4. **스키마가 앞서 있는 항목** — 전화번호 기반 차단, 전역 알림 SSE, 장소 추천은 MVP에 억지로 포함시키면 일정이 흔들린다.

---

## 11. 추천되는 합리적 MVP 3차 최종 범위

MVP 3차는 **추천 3명 / 호감 수락 기반 채팅 / 댓글 기반 피드 상호작용 / 공통 차단·신고**까지를 1차 출시 범위로 정의한다.

장소 추천, 전역 알림 스트림, 전화번호 기반 차단, 운영용 랭킹 / 재심사 / 관리 도구는 모두 **후속 범위로 분리**한다.

> MVP 3차는 **지금 구현해서 실제 사용자가 행동할 수 있는, 최소하지만 닫힌 서비스 루프**를 만드는 단계다.
>
> 추천으로 시작해 채팅이나 피드 댓글로 이어지고, 차단 / 신고가 공통 정책으로 작동하며, 프로필과 추천 설정을 통해 다음 액션을 다시 만드는 구조만 우선 출시한다.
