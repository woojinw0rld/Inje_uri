# 04. 인증 관련 문제

## 문서 범위

이 문서는 인제우리 프로젝트 문제 분석 중 **페이지 접근 인증, 실제 세션 검증, 회원가입 사전 인증, 세션/쿠키 기준 불일치**에 관련된 문제만 분리한 문서입니다.

포함 범위:

- middleware 인증
- 세션 쿠키
- 회원가입 사전 인증
- 생년월일 검증
- 온보딩 이후 추천 생성 위치
- 인증/인가 에러 코드

제외 범위:

- 호감/매칭
- 채팅방/채팅
- feed/댓글/신고/차단

---

## 1. 인증 미들웨어가 실제 세션이 아니라 고정 쿠키를 봄

### 심각도

**높음**

### 문제 요약

페이지 라우팅 보호 미들웨어가 실제 세션 쿠키가 아니라 `injeuri_bus_auth=1`이라는 고정 쿠키를 확인합니다.

API는 `getAuthUser()`를 통해 실제 세션을 검증하지만, 페이지 접근 미들웨어는 별도 고정 쿠키를 기준으로 판단합니다.

---

### 미들웨어 위치

파일:

```text
src/middleware.ts
```

관련 위치:

```text
3: APP_AUTH_COOKIE_NAME, APP_AUTH_COOKIE_VALUE import
7: injeuri_bus_auth === "1" 확인
```

현재 코드:

```ts
import { APP_AUTH_COOKIE_NAME, APP_AUTH_COOKIE_VALUE, isProtectedAppPath } from '@/lib/auth/constants';

const isAuthenticated = request.cookies.get(APP_AUTH_COOKIE_NAME)?.value === APP_AUTH_COOKIE_VALUE;
```

---

### 고정 쿠키 상수 위치

파일:

```text
src/lib/auth/constants.ts
```

위치:

```text
4-6
```

현재 코드:

```ts
export const APP_AUTH_COOKIE_NAME = 'injeuri_bus_auth';
export const APP_AUTH_COOKIE_VALUE = '1';
export const APP_AUTH_COOKIE_MAX_AGE_SECONDS = 60 * 60 * 12;
```

---

### 실제 서버 세션 위치

파일:

```text
src/server/lib/auth.ts
```

관련 위치:

```text
26: SESSION_COOKIE_NAME = "injeuri_session"
120-129: attachSessionCookie
252-279: resolveCurrentUser
290-294: getAuthUser
```

현재 실제 세션 쿠키:

```ts
export const SESSION_COOKIE_NAME = 'injeuri_session';
```

---

### 영향

- 화면 라우팅 보호와 API 인증 기준이 다름
- 사용자가 `injeuri_bus_auth=1` 쿠키만 만들면 보호 페이지 화면에 접근할 가능성이 있음
- 프론트가 mock 데이터를 사용하는 화면에서는 실제 인증 없이도 앱이 동작하는 것처럼 보일 수 있음
- “학교 인증된 사용자만 이용”이라는 서비스 전제와 어긋남

---

### 권장 수정

`middleware.ts`에서는 최소한 `injeuri_session` 존재 여부를 기준으로 바꿉니다.

예시:

```ts
const isAuthenticated = request.cookies.has('injeuri_session');
```

다만 이것도 완전한 세션 검증은 아닙니다.  
권장 구조는 다음과 같습니다.

- middleware: 세션 쿠키 존재 여부 기반의 얕은 라우팅 보호
- server layout/API: 실제 세션 검증
- API 요청: 항상 `getAuthUser()` 기준 권한 확인
- 보호 페이지: 서버에서 현재 사용자 조회 실패 시 login으로 redirect

## 2. 회원가입 시 사전 인증 생년월일과 입력 생년월일을 비교하지 않음

### 심각도

**높음**

### 문제 요약

사전 인증에서는 생년월일 해시를 저장하지만, 회원가입 시 사용자가 입력한 생년월일과 사전 인증 생년월일을 비교하지 않습니다.

학번은 사전 인증된 값으로 저장되지만, 생년월일은 회원가입 폼 입력값이 그대로 저장될 수 있습니다.

---

### 사전 인증 저장 위치

파일:

```text
src/server/lib/auth.ts
```

위치:

```text
184-198: issuePreSignupVerification
187: birthHash 생성
191-196: pre_signup_verification 저장
```

현재 코드:

```ts
const birthHash = hashBirth(birth);

await createPreSignupVerification({
  tokenHash,
  studentNumber,
  birthHash,
  expiresAt,
});
```

---

### 회원가입 위치

파일:

```text
src/server/services/auth/auth.service.ts
```

위치:

```text
138-142: preSignup 조회
167-182: 유저 생성
174-175: 사용자가 입력한 input.birth 저장
181: 학번은 preSignup.studentNumber 사용
```

현재 구조:

```ts
export async function register(input: RegisterInput, preSignupToken: string | null) {
  const preSignup = await consumePreSignupVerificationToken(preSignupToken);

  const passwordHash = await bcrypt.hash(input.password, 10);

  await createUser({
    birth: input.birth,
    birth_hash: hashBirth(input.birth),
    student_number: preSignup.studentNumber,
  });
}
```

문제는 `preSignup.birthHash`와 `hashBirth(input.birth)`를 비교하는 코드가 없다는 점입니다.

---

### 영향

- 사전 인증과 회원가입 정보가 불일치할 수 있음
- 학번은 인증된 사용자지만 생년월일은 임의 입력값으로 저장될 수 있음
- 이후 생년월일 기반 본인확인, 나이 검증, 계정 복구 정책이 흔들릴 수 있음

---

### 권장 수정

회원가입 직전에 다음 검증을 추가합니다.

```ts
if (preSignup.birthHash !== hashBirth(input.birth)) {
  throw new ApiError(ERROR.INVALID_VERIFICATION, "인증 정보와 생년월일이 일치하지 않습니다.");
}
```

또는 기존 에러 체계에 맞춰 `AppError`/`ApiError` 중 하나로 통일합니다.

## 3. 인증 기준이 화면과 API에서 분리되어 있음

### 심각도

**높음**

### 문제 요약

페이지 접근 보호는 `src/middleware.ts`의 고정 쿠키 기준이고, API 접근 보호는 `src/server/lib/auth.ts`의 실제 세션 기준입니다.

즉, 사용자 입장에서 화면은 들어왔는데 API는 401을 반환하는 상태가 생길 수 있습니다.

---

### 화면 보호 기준

파일:

```text
src/middleware.ts
```

현재 기준:

```text
injeuri_bus_auth=1
```

---

### API 보호 기준

파일:

```text
src/server/lib/auth.ts
```

현재 기준:

```text
injeuri_session
```

사용 예시:

```text
src/app/api/reports/route.ts
src/app/api/blocks/route.ts
src/app/api/chat-room/[id]/messages/route.ts
src/app/api/interests/send/route.ts
src/app/api/interests/received/route.ts
```

각 API에서 대체로 다음 패턴을 사용합니다.

```ts
const user = await getAuthUser(request);
if (!user) return fail("UNAUTHORIZED", "인증이 필요합니다.");
```

---

### 영향

- 인증이 안 된 사용자가 화면에 접근할 수 있음
- 화면은 열리지만 API 요청은 실패하는 UX 발생
- mock 데이터가 있는 화면에서는 인증 실패가 숨겨질 수 있음
- QA 단계에서 인증 문제가 늦게 발견될 수 있음

---

### 권장 수정

1. `injeuri_bus_auth` 쿠키 제거 또는 사용 범위 명확화
2. route protection 기준을 `injeuri_session`으로 통일
3. 서버 컴포넌트 또는 layout에서 current user fetch 추가
4. 인증 실패 시 login redirect
5. mock 데이터 화면 제거 후 API 401 상태를 UI에서 처리

---

## 5. 인증 관련 에러 코드 정리 필요

### 심각도

**중간**

### 문제 요약

API별로 `UNAUTHORIZED`, `FORBIDDEN`, `INVALID_*` 에러를 사용하고 있지만, 인증/인가 에러 코드가 전역으로 완전히 정리되어 있는지 확인이 필요합니다.

기존 커뮤니케이션에 따르면 다음 에러 코드 방향이 제안되어 있습니다.

```ts
export const ERROR = {
  UNAUTHORIZED: "UNAUTHORIZED",
  FORBIDDEN: "FORBIDDEN",
  NOT_FOUND: "NOT_FOUND",
  BLOCKED_RELATION: "BLOCKED_RELATION",
  ROOM_EXPIRED: "ROOM_EXPIRED",
  ROOM_NOT_ACTIVE: "ROOM_NOT_ACTIVE",
  DUPLICATE_ACTIVE_ROOM: "DUPLICATE_ACTIVE_ROOM",
  REMATCH_TOO_SOON: "REMATCH_TOO_SOON",
  INVALID_SOURCE: "INVALID_SOURCE",
} as const;
```

---

### 권장 수정

인증 관련 에러 코드는 최소한 다음 기준으로 통일합니다.

```text
UNAUTHORIZED: 로그인 안 됨
FORBIDDEN: 로그인은 됐지만 리소스 접근 권한 없음
INVALID_VERIFICATION: 사전 인증 정보 불일치
SESSION_EXPIRED: 세션 만료
```

응답 포맷은 다음과 같이 통일합니다.

```ts
{
  success: false,
  error: {
    code: string,
    message: string
  }
}
```

---

## 우선 수정 순서

1. `src/middleware.ts`에서 `injeuri_bus_auth` 의존 제거
2. route protection 기준을 `injeuri_session`으로 통일
3. 회원가입 시 `preSignup.birthHash`와 `hashBirth(input.birth)` 비교 추가
4. 온보딩 완료 API에서 추천 생성 호출 위치 확정

---

## 최종 판단

인증 영역의 핵심 문제는 “로그인 API가 있느냐”가 아니라, **화면 접근 기준과 API 접근 기준이 다르다**는 점입니다.

가장 먼저 고쳐야 할 항목은 다음입니다.

1. `injeuri_bus_auth=1` 기반 미들웨어 제거
2. 실제 세션 쿠키 `injeuri_session` 기준으로 route protection 통일
3. 사전 인증 생년월일과 회원가입 생년월일 일치 검증 추가
